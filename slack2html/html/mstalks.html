jongold
2020-11-17 20:56
Log write throughput is practically unlimited in the datacenter, by using RAID style arrays of managed disks (per VM), combined with the adaptive batching mechanisms. We easily get many GB of throughput/s from a single VM in configurations like these. The bottleneck is typically the time it takes to run actual code. This is what I meant by log activity is request bound rather than state change bound. 1GB of incoming requests results in 1GB of log writing, regardless of how much state is changed by executing the request. With cost being limited these days by CPU, the extra log movement and writing is very cheap. Even the cost of contention associated with writing to the tail of the log is typically very dominated by deserialization and request execution time in practice.

jongold
2020-11-17 21:00
In terms of latency, we are limited by the time to make a request durable in a replicated environment. Today, this latency is 100s of microseconds within the datacenter. At the time the paper was written, that storage service hadn't been made available yet. I suspect we could get under a millisecond with the latest. Across datacenters, of course, will be significantly larger, although we haven't measured.

jongold
2020-11-17 21:02
I just realized that there may have been a misunderstanding in the talk about the log. Communicating immortals no NOT share a log. They each have their own independent logs. So if you manually sharded an easily partitioned service, each shard would get it's own log, and be coordination free.

jongold
2020-11-17 21:10
Orleans doesn't have virtual resiliency today. Logically failure free execution can be guaranteed in a manner similar to what I described for Azure functions in the talk. For instance, application writers must explicitly save and restore state, must have their own logic (e.g. using seq nos) to guarantee logically exactly once execution, and generally write their code with partial executions in mind. I believe it does have a transaction mechanism, though, which can be used to atomically commit large blocks of code, which helps with writing code against partial executions, although at high cost, and still requires developer intervention.

jongold
2020-11-17 21:11
Ambrosia covers both logical exactly once execution, and state recovery, which is a minimum for supporting virtual resiliency.

jongold
2020-11-17 21:12
It also automatically recovers all connections to other Ambrosia immortals, even if the IP address, or operating system, or CPU changes.
